# KTOX SDR Suite Design

## Goal

Add a HackRF-focused SDR Suite to KTOX_Pi as a separate service on port 8081, linked from the existing KTOX WebUI without changing the main WebUI server architecture.

## Scope

The first build provides:

- HackRF device status through `hackrf_info`.
- Frequency sweep support through `hackrf_sweep`.
- IQ capture support through `hackrf_transfer`.
- FFT, waterfall row generation, power spectrum normalization, and peak detection helpers.
- SQLite capture indexing under `captures/index.db`.
- A standalone Flask + SocketIO server at `services/sdr_server.py`.
- A standalone browser UI under `static/sdr/`.
- A repo-local systemd unit template under `scripts/ktox-sdr.service`.
- A small KTOX WebUI link to open the SDR Suite on `:8081`.

This build does not directly install packages, enable systemd services, or write to `/etc` from the development workstation.

## Architecture

The SDR Suite runs independently from `web_server.py` so HackRF subprocesses, streaming threads, capture files, and SocketIO state cannot destabilize the main KTOX WebUI. The main WebUI gets only a navigation link that derives the current hostname and opens `http://<host>:8081`.

The backend is split into focused modules:

- `sdr/device.py` wraps HackRF CLI commands with safe subprocess handling.
- `sdr/processing.py` owns hardware-independent signal math.
- `sdr/database.py` owns the SQLite capture index.
- `sdr/handlers.py` owns capture naming, presets, file metadata, and sweep parsing helpers.
- `services/sdr_server.py` owns HTTP routes, SocketIO events, background jobs, and serving the SDR UI.

The frontend is a small single-page app:

- `static/sdr/index.html` defines Dashboard, Waterfall, Sweep, Capture, and Settings tabs.
- `static/sdr/js/api.js` wraps fetch calls.
- `static/sdr/js/waterfall.js` owns canvas drawing.
- `static/sdr/js/app.js` owns UI state, SocketIO connection, settings persistence, and event binding.
- `static/sdr/css/style.css` provides a KTOX-compatible dark UI.

## Data Flow

Device info:

1. Browser calls `GET /api/hackrf/info`.
2. `sdr_server.py` calls `HackRFManager.get_info()`.
3. `device.py` runs `hackrf_info`, parses useful fields, and returns JSON.

Capture:

1. Browser posts frequency, sample rate, gain, and duration to `POST /api/hackrf/capture`.
2. The server schedules a background job.
3. `handlers.py` creates a safe filename under `captures/`.
4. `device.py` runs `hackrf_transfer` and tracks only the child process it started.
5. `database.py` indexes the resulting file metadata.

Waterfall:

1. Browser emits `start_waterfall` over SocketIO.
2. The server starts a per-client background worker.
3. The worker runs bounded HackRF reads where hardware is available.
4. FFT rows are generated through `processing.py` and emitted as compact JSON arrays.
5. The browser draws rows onto a canvas.

Frequency sweep:

1. Browser posts start, stop, bin width, and dwell values.
2. The server schedules a background sweep.
3. `device.py` runs `hackrf_sweep`.
4. `handlers.py` parses sweep output into structured rows for the UI.

## Error Handling

The backend returns structured JSON errors for missing HackRF tools, missing devices, invalid request parameters, command timeouts, and failed captures. Long-running jobs are limited by a small `ThreadPoolExecutor` and do not use broad process cleanup. Stop actions terminate only the process object owned by the active worker.

The UI shows unavailable hardware clearly and keeps offline/demo display behavior limited to frontend rendering state. It does not claim a HackRF is connected unless `/api/hackrf/info` succeeds.

## Security And Safety

Inputs are range checked before command execution. Paths for captures are generated by the application or resolved under the `captures/` root. Delete/download APIs do not accept arbitrary absolute paths. Backend command invocation uses argument arrays instead of shell strings.

Transmit support is represented in `HackRFManager` as a controlled method but is not exposed in the first UI. The first UI focuses on receive, sweep, and capture workflows.

## Testing

Automated tests should cover:

- HackRF command construction and `hackrf_info` parsing using injected command runners.
- FFT normalization, waterfall row shape, and peak detection.
- SQLite schema creation, insert, list, and delete metadata operations.
- Capture filename safety and frequency preset data.
- Flask endpoint behavior with mocked manager methods.

Manual Pi/Kali validation remains required for:

- `hackrf_info` with real hardware.
- Real IQ capture to `captures/`.
- Real waterfall streaming.
- Service installation through the provided systemd template.

## Acceptance Criteria

- The SDR Suite can run with `python3 services/sdr_server.py` and serve `http://localhost:8081`.
- `GET /api/hackrf/info` returns JSON success or a clear unavailable error.
- `POST /api/hackrf/capture` records job metadata and indexes successful files.
- `GET /api/hackrf/captures` lists indexed captures.
- SocketIO waterfall start/stop events do not block the main server thread.
- The main KTOX WebUI includes an SDR Suite entry that opens the separate service.
- Pure Python validators pass in this workspace without requiring HackRF hardware.
